Strongly typed API

The tools and languages we use on the Web today weren’t made for developing complex applications. Let’s take a leaf out of the book of desktop GUI development, and upgrade our workflow.

Typescene is written in TypeScript. Just like other strongly typed languages such as C++, C# and Java, it detects errors in your code at compile-time, and can help you work faster with advanced IDE support.

Get going quickly. Get started quickly without having to make trivial decisions about the architecture and dependencies of a simple app. Lay out your screens or dialogs, wire up event handlers in your view (component) classes, and use whatever method suits your project best to update the page using the UI module’s API.

Scale up to big projects. Leverage the power of TypeScript to maintain large code bases. Use the App module to organize your code by features and activities, independent of your business logic and data models.

3

All Open Source

All the code for Typescene is available on GitHub, and distributed through NPM.

Typescene has no dependencies. Other than TypeScript, no external packages are required to compile the Typescene code, or an application that uses it. Modules are produced in standard CommonJS and ES6 format, meaning any JS packager can package your application for deployment in production.

Fewer dependencies means fewer things that can break when you least expect them to.

Install and import: install the root module from NPM into your Node.js project. 

npm install @typescene/typescene

Then, depending on your deployment scenario, either copy the typescene.min.js file to your assets folder, or use a tool such as Webpack to integrate the Typescene sources into your build.

Getting Started

The Typescene API has been put together carefully to cover as many scenarios as possible with only a few key constructs. To avoid getting lost later on, start with these concepts:



Modules and Components
Organize your code using modules to break up your app into separate files. Each module exports its own external API, and hides its implementation from the other modules.

When you break up your User Interface into separate pieces, each representing a part of the screen to update, we call those components. One module may contain several component classes that work together.

Typescene itself consists of 3 main modules: Async, UI, and App, each with a different purpose and their own external API.



@Typescene/Async
The ‘Async’ library API is used throughout the entire toolkit. It provides simple but powerful ways to model asynchronous behavior. Ultimately, this enables the GUI code to start rendering parts of your app without having all your data upfront, and then fill in the gaps later.

The most well-known construct is the Promise object, which can be passed around as a reference to a future value. Typescene makes heavy use of this, but also includes Signals (event emitters), and observable values and arrays.

Take a look at some examples to get an understanding of how this works.



@Typescene/UI
Almost all of the classes exported by the ‘UI’ module are components. These can be put together to build forms, cards, dialogs, pages, etc.

In broad strokes, components include container level components, which only contain block level components, stacked top-to-bottom. Types of blocks include tables, lists, cards, and rows—which in turn only contain control level components. Controls may be buttons, text fields, labels, etc.

You can choose to build a component with any of these classes as a starting point, but usually a container is a solid choice.

Read more about how to get started with the UI module.



@Typescene/App
This is an optional capstone module that provides a way to break down your code into larger scale Activities.

Unless your application has only one view that is always visible, you may want to keep track of which views are currently active, and the state that these views are in. The App module makes this possible by letting you define Activity objects that can be started and suspended on a last-in-first-out basis (i.e. the history stack).

An Activity class can also register itself as the main handler for a route (i.e. last part of the URL), and a Component can then register itself as the main view to be displayed when that activity is on top of the activity stack. At any time, the Application object can provide an observable reference to the currently active instance of some activity class, to allow other code to access the current overall state.

This allows for very clean code, where lower-level view code (e.g. for the text to be displayed on a label) only depends on higher-level activity code, not the other way around. Down the line, components can easily observe application state without forcing developers to learn complex paradigms such as functional or reactive programming.

Go to the App module docs to read more about application architecture.
